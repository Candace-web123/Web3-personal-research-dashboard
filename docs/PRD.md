## 一、产品定位

### 1.1 产品名称

暂定名称：**Web3 个人投研决策支持系统**

### 1.2 一句话定位

一个面向个人投资者的 Web3 投研 Dashboard：  
**打开即可看到今日市场环境、BTC 周期位置、ETH / 主流资产状态、稳定币流动性、30 币种池异动、强链强赛道、强协议、Alpha 观察池、仓位建议与复盘问题。**

### 1.3 产品不是做什么

本产品不是：

- 喊单工具
- 自动交易工具
- 保证收益工具
- KOL 热点跟踪器
- 单纯行情看板
- 30 个币平均展示的行情站

本产品要做的是：

- 让用户知道现在市场处于什么状态
- 让用户知道资金流向哪里
- 让用户知道 30 个币种池中哪些资产出现真实异动
- 让用户知道哪些链、赛道、协议正在变强
- 让用户判断协议强是否能传导到代币强
- 让用户建立观察池，而不是情绪化追涨
- 让用户根据 BTC 周期和市场环境调整仓位

## 四、模块一：BTC 周期位置判断系统

### 4.1 模块目标

回答三个问题：

1. 现在 BTC 处于什么周期位置？
2. 当前适合分批买入、持有、减仓，还是观望？
3. BTC 周期位置是否支持小仓位观察山寨 Alpha？

### 4.2 核心指标

| 指标 | 数据源 | 作用 |
|---|---|---|
| BTC 当前价格 | CoinGecko / TradingView | 判断当前市场价格 |
| MVRV Z-Score | Glassnode / LookIntoBitcoin | 判断顶部与底部风险 |
| NUPL | Glassnode / LookIntoBitcoin | 判断市场盈利状态与情绪阶段 |
| Puell Multiple | LookIntoBitcoin | 判断矿工收入周期 |
| Pi Cycle Top | LookIntoBitcoin | 辅助判断历史顶部风险 |
| 200 周均线 | TradingView / LookIntoBitcoin | 判断长期支撑与周期底部区域 |
| 恐惧贪婪指数 | Alternative.me | 判断市场情绪是否极端 |
| BTC ETF 净流入 | SoSoValue / Farside | 判断传统资金是否流入 BTC |
| 美元指数 DXY | Trading Economics / Investing | 判断宏观流动性压力 |
| 10年期美债收益率 | Trading Economics / Investing | 判断风险资产压力 |
| 美联储利率与降息预期 | CME FedWatch / Investing | 判断宏观流动性方向 |

### 4.3 BTC 周期状态分类

| 周期状态 | 典型特征 | 操作倾向 |
|---|---|---|
| 底部积累区 | MVRV 低位、NUPL 恐慌、价格接近或低于200周均线、市场恐惧 | 分批买入 BTC |
| 上涨早期 | BTC 站上长期均线，ETF/稳定币开始流入，情绪从恐惧修复 | 定投 / 分批加仓 |
| 上涨中期 | 链上指标健康，价格趋势向上，市场情绪未极端 | 持有为主，少量观察 Alpha |
| 过热区 | 恐惧贪婪高位、MVRV/NUPL偏高、杠杆升温 | 控制加仓，降低追高 |
| 顶部风险区 | 多个顶部指标共振，情绪极端贪婪，ETF/稳定币流入放缓 | 分批减仓 |
| 下跌/防守区 | BTC 跌破关键趋势，稳定币收缩，TVL/DEX同步走弱 | 稳定币为主，等待新信号 |

## 四-1、模块补充：多资产观察池管理系统

### 4-1.1 模块目标

本模块用于解决一个问题：

> BTC 负责判断大周期，但用户每天实际会看到很多主流币、稳定币和热点币。如果系统只展示 BTC，会显得太窄；如果系统平均分析 30 个币，又会变成行情站。

因此系统采用“1 个周期锚点 + 30 个可配置观察币池”的结构：

- BTC：作为周期锚点，用于判断市场大方向和风险偏好。
- ETH：作为主流风险资产参考，用于判断主流资金是否继续愿意承担风险。
- 稳定币：作为场内资金水位指标，用于判断市场是否有新增流动性。
- 主流币：作为市场轮动观察对象，用于识别资金正在流向哪些方向。
- 热点币 / Alpha 候选：作为观察池来源，不直接等于买入清单。

### 4-1.2 产品边界

多资产观察池不是让用户每天研究 30 个币，也不是鼓励用户分散买入 30 个币。

本模块只做三件事：

1. 建立一个可维护的币种观察宇宙。
2. 每天扫描哪些资产出现真实异动。
3. 将少数满足条件的资产推进 Alpha 观察池或深挖池。

### 4-1.3 默认币种池分层

| 层级 | 资产类型 | 作用 | 示例 |
|---|---|---|---|
| S 层 | 周期锚点资产 | 判断大盘周期和风险偏好 | BTC |
| A 层 | 核心主流资产 | 判断主流资金方向 | ETH、SOL、BNB |
| B 层 | 稳定币 / 流动性资产 | 判断场内资金水位，不作为普通买入标的 | USDT、USDC、DAI / USDS、USDe |
| C 层 | 主流板块资产 | 判断赛道轮动和资金偏好 | LINK、AVAX、SUI、APT、ARB、OP、TON、XRP、DOGE 等 |
| D 层 | 热点 / Alpha 候选资产 | 作为观察池来源，需要强筛选 | ONDO、PENDLE、ENA、JUP、AAVE、UNI、HYPE 等 |

说明：

- 稳定币不和 BTC / ETH / 山寨币放在同一评分体系中。
- 稳定币主要用于判断市场资金水位，而不是作为 Alpha 标的。
- 热点币只有在数据、估值、供给、催化和风险同时通过筛选后，才可进入 Alpha 观察池。

### 4-1.4 30 币种池的 MVP 规则

MVP 阶段支持约 30 个币种的数据展示和异动扫描，但不对 30 个币全部做深度投研。

30 个币在 MVP 中分为三类：

| 类型 | 处理方式 | 输出位置 |
|---|---|---|
| 必看资产 | 每日固定展示 | BTC、ETH、稳定币总市值 |
| 扫描资产 | 每日扫描价格、成交量、市值、链/赛道归属、解锁风险 | 主流币异动 Top 5 |
| 观察资产 | 只有触发异动或评分达标时才进入观察池 | Alpha 观察池 Top 10 |

系统首页只展示：

1. BTC 周期判断
2. ETH / 主流资产状态
3. 稳定币流动性趋势
4. 今日主流币异动 Top 5
5. Alpha 观察池 Top 10
6. 今日重点深挖项目 1-3 个

### 4-1.5 币种进入观察池的条件

一个币种不能因为“涨得多”直接进入 Alpha 观察池。

进入 Alpha 观察池至少需要满足以下条件中的 3 项：

| 条件 | 说明 |
|---|---|
| 所属链变强 | 链上 TVL、Stablecoins、DEX Volume、Fees 至少两项改善 |
| 所属赛道变强 | 同赛道多个项目同步改善，而不是单项目孤立上涨 |
| 协议数据变强 | TVL、Volume、Fees、Revenue、Active Users 等连续改善 |
| 代币价值传导更清晰 | 有 staking、buyback、burn、fee sharing、使用需求或治理升级预期 |
| 估值没有明显透支 | Market Cap / FDV / Revenue / TVL 与同赛道相比未明显过高 |
| 供给压力可控 | 未来 30 / 90 / 180 天无重大解锁或通胀压力 |
| 有明确催化 | 产品上线、生态激励、上币、协议升级、政策或资金催化 |
| 市场环境支持 | 今日市场不是谨慎或防守，或仅作为观察不新增仓位 |

### 4-1.6 币种状态规则

| 状态 | 含义 | 系统动作 |
|---|---|---|
| 普通观察 | 在 30 币种池中，但无明显异动 | 不进入首页重点展示 |
| 异动观察 | 价格 / 成交量 / 数据有变化，但未验证 | 展示在主流币异动 Top 5 |
| Alpha 候选 | 数据和叙事初步匹配 | 进入 Alpha 观察池 |
| 重点深挖 | 多项数据共振，风险可控 | 进入今日深挖项目 |
| 风险升高 | 价格先行、解锁临近、数据转弱或估值透支 | 降低评级或移除 |
| 移除 | 原始逻辑失效或风险不可控 | 从观察池移除 |

### 4-1.7 多资产扫描输出格式

【今日多资产扫描】

- BTC 周期锚点：
- ETH / 主流资产状态：
- 稳定币流动性：
- 今日主流币异动 Top 5：
- 今日进入 Alpha 观察池：
- 今日风险升高资产：
- 今日最值得深挖项目：
- 今日不建议追高资产：
- 明日继续观察：

## 五、模块二：每日市场环境判断系统

### 5.1 模块目标

回答：

1. 今天市场是强进攻、中性轮动、谨慎还是防守？
2. 今天适合看什么，不适合做什么？
3. 今天是否适合新增 Alpha 观察？

### 5.2 指标与评分

| 指标 | 观察内容 | 评分 |
|---|---|---:|
| Stablecoins | 总市值、1D/7D/30D变化、USDT Dominance | 偏多 +1 / 中性 0 / 偏空 -1 |
| DeFi TVL | 总 TVL、24h/7D/30D趋势 | 偏多 +1 / 中性 0 / 偏空 -1 |
| DEX Volume | 24h成交量、7D变化 | 偏多 +1 / 中性 0 / 偏空 -1 |
| ETF / DAT | BTC/ETH ETF、DAT资金流 | 偏多 +1 / 中性 0 / 偏空 -1 |
| 宏观政策 | CPI、PCE、FOMC、美元指数、美债收益率、监管事件 | 偏多 +1 / 中性 0 / 偏空 -1 |

### 5.3 环境分类

| 总分 | 市场环境 | 解释 |
|---:|---|---|
| +3 到 +5 | 强进攻 | 资金、交易、宏观多项共振，适合看强势资产 |
| 0 到 +2 | 中性轮动 | 局部赛道和强链机会，避免无脑追高 |
| -1 到 -2 | 谨慎 | 只观察，降低新增高风险仓位 |
| -3 到 -5 | 防守 | 稳定币和复盘为主，不主动进攻 |

## 六、模块三：强链筛选系统

### 6.1 判断逻辑

强链不是只看 TVL 最大，而是看：

1. TVL 是否增长
2. Stablecoins 是否有规模或增长
3. DEX Volume 是否活跃
4. Perps 是否升温
5. Fees / Revenue 是否有真实使用
6. Bridge Netflow 是否有资金流入
7. 1D、7D、1M 是否趋势一致
8. 是否只是单日 spike

### 6.2 强链类型

| 类型 | 特征 | 适合观察 |
|---|---|---|
| 资金沉淀型 | TVL、稳定币规模大且稳定 | Lending、RWA、Stablecoin、LSDFi |
| 交易活跃型 | DEX / Perps 成交量强 | DEX、Perps、Meme、Launchpad |
| 应用收入型 | Fees / Revenue 增长明显 | 高频使用型协议 |
| 激励驱动型 | TVL涨但收入弱 | 谨慎，防止补贴结束后撤退 |

## 八、模块五：强协议筛选系统

### 8.1 核心判断

强协议不等于强代币。系统必须区分：

- TVL 高 ≠ 代币低估
- Fees 高 ≠ Revenue 高
- Revenue 高 ≠ Token Holder Revenue 高
- 协议赚钱 ≠ 持币人赚钱
- 社媒热度 ≠ 数据验证
- 鲸鱼买入 ≠ 必然利好

### 8.2 协议观察字段

| 字段 | 说明 |
|---|---|
| 协议名称 | 项目名称 |
| 所属链 | Ethereum / Solana / Base 等 |
| 所属赛道 | Lending / DEX / Perps 等 |
| TVL | 资金沉淀 |
| 1D/7D/1M TVL变化 | 趋势强弱 |
| Fees | 用户是否真实付费 |
| Revenue | 协议是否留存收入 |
| Holder Revenue | 持币人是否受益 |
| Active Users / Loans / Volume | 真实业务指标 |
| Market Cap | 当前市值 |
| FDV | 完全稀释估值 |
| Mcap/TVL | 估值参考 |
| FDV/Revenue | 收入估值参考 |
| 解锁压力 | 未来30/90/180天供给 |
| 代币价值传导 | 协议强是否能传导到代币 |

## 九、模块六：Alpha 标的筛选雷达

### 9.1 模块目标

不是找“涨得最多的币”，而是找：

- 数据开始变强
- 所属链或赛道正在变强
- 协议增长真实
- 代币能捕获价值
- 估值没有明显透支
- 供给和解锁压力可控
- 未来仍有催化
- 风险可以被跟踪

### 9.2 Alpha 评分模型

| 维度 | 核心问题 | 评分 |
|---|---|---:|
| 市场环境 | 今天适合进攻、轮动、谨慎还是防守？ | -2 到 +2 |
| 所属链 | 这条链是否有资金流入、交易活跃或真实使用？ | 0-10 |
| 所属赛道 | 赛道是否多项目共振？ | 0-10 |
| 协议数据 | TVL / Volume / Fees / Revenue 是否持续增长？ | 0-10 |
| 收入质量 | 收入来自真实用户，还是补贴和刷量？ | 0-10 |
| 代币价值传导 | 协议变强是否能让代币受益？ | 0-10 |
| 估值合理性 | Mcap / FDV / Revenue / TVL 是否透支？ | 0-10 |
| 供给结构 | 流通比例、解锁、通胀是否可控？ | 0-10 |
| 流动性结构 | CEX / DEX 深度、成交量是否健康？ | 0-10 |
| 催化因素 | 未来1-3个月是否还有催化？ | 0-10 |
| 风险扣分 | 是否有一票否决风险？ | -10 到 0 |

### 9.2.1 代币价值传导 MVP 简化规则

本产品必须持续区分“强协议”和“强代币”。协议 TVL、Fees、Revenue 变强，只能说明协议本身有业务变化，不代表代币一定能捕获价值。

V1.2 MVP 不直接采用复杂公式，而先采用人工标签 + 简化评分，避免因数据缺失导致伪精确。

| 字段 | 选项 | 说明 |
|---|---|---|
| 传导类型 | 现金流型 / 回购型 / 使用型 / 治理预期型 / 无传导 | 判断协议增长如何传导到代币 |
| 传导强度 | 强 / 中 / 弱 / 无 | 作为 Alpha 评分中的关键因子 |
| 数据依据 | Holder Revenue / buyback / burn / staking / gas / 抵押需求 / 治理预期 | 必须写明判断依据 |
| 是否影响 Alpha 评级 | 是 / 否 | 传导弱或无传导时，默认不得评为 A |
| 人工备注 | 自由文本 | 记录为什么认为该代币能或不能受益 |

处理规则：

1. **无传导**：即使协议数据强，也不得进入 A 级，只能作为观察或降级处理。
2. **治理预期型**：若没有明确收入、回购、销毁、质押或使用需求，只能给中低传导分。
3. **现金流型 / 回购型 / 使用型**：需记录数据来源，避免只凭项目方叙事判断。
4. **数据缺失**：标记为“不确定”，不得用乐观假设替代事实。

### 9.3 Alpha 评级

| 分数 | 评级 | 处理方式 |
|---:|---|---|
| 80+ | A | 重点研究，持续跟踪，等待合适价格 |
| 65-79 | B | 进入观察池，等待数据确认或回调 |
| 50-64 | C | 只观察，不急 |
| 50以下 | D | 暂时排除 |

### 9.4 Alpha 观察池输出

| 排名 | 项目 | Token | 链 | 赛道 | 核心异动 | 代币传导 | 估值/供给 | 催化 | 最大风险 | 综合分 | 评级 | 处理方式 |
|---|---|---|---|---|---|---|---|---|---|---:|---|---|

每个项目必须回答：

1. 为什么进入观察池？
2. 它是强协议 + 强代币，还是强协议 + 弱代币？
3. 当前价格是否已经透支数据？
4. 未来30/90/180天解锁压力如何？
5. 哪个数据变化会让我升级？
6. 哪个风险出现会让我排除？

## 十、模块七：历史观察池跟踪与升级/降级系统

### 10.1 模块目标

解决一个核心问题：

> 不能每天只新增 Alpha 项目，而不复盘旧项目。

系统需要每天回看过去加入观察池的项目，判断它们是否继续满足观察理由。

### 10.2 跟踪对象

| 对象 | 跟踪频率 |
|---|---|
| A 类项目 | 每日跟踪 |
| B 类项目 | 每 2-3 日跟踪 |
| C 类项目 | 每周跟踪 |
| D 类项目 | 不跟踪或仅事件触发时查看 |

### 10.3 跟踪字段

| 字段 | 说明 |
|---|---|
| 加入观察池日期 | 记录什么时候开始观察 |
| 加入原因 | 当时为什么进入观察池 |
| 初始价格 | 加入当天价格 |
| 当前价格 | 当前价格 |
| 价格变化 | 价格是否已经大幅反映 |
| 初始 TVL / 当前 TVL | 判断协议数据是否继续变强 |
| 初始 Fees / 当前 Fees | 判断收入是否继续变强 |
| 初始 Revenue / 当前 Revenue | 判断商业模式是否持续 |
| 初始 Holder Revenue / 当前 Holder Revenue | 判断代币持有人是否受益 |
| 初始评级 / 当前评级 | 判断是否升级或降级 |
| 推翻逻辑是否触发 | 判断是否移除 |
| 下一步验证 | 明天或下周继续看什么 |

### 10.4 升级规则

项目满足以下条件，可升级：

- 所属链继续变强
- 所属赛道继续变强
- 协议 TVL / Volume / Fees / Revenue 持续改善
- 代币价值传导更加清晰
- 估值没有大幅透支
- 解锁风险下降
- 有新的明确催化

### 10.5 降级规则

项目出现以下情况，应降级：

- 协议数据连续走弱
- 价格上涨但基本面没有跟上
- Fees 高但 Revenue 或 Holder Revenue 没有改善
- 解锁压力加大
- 叙事热度下降且数据无法支撑
- 所属链或赛道走弱
- 出现安全、监管、团队、流动性风险

### 10.6 移除规则

出现以下情况，应从观察池移除：

- 原始加入理由失效
- 数据被证明是补贴或刷量
- 代币没有价值捕获，且无改善预期
- 30/90/180天供给压力过大
- 发生合约安全、监管或团队信任危机
- 流动性不足，无法正常进出

### 10.7 输出格式

【历史观察池跟踪】

| 项目 | 加入日期 | 初始评级 | 当前评级 | 数据变化 | 价格变化 | 是否验证原逻辑 | 动作 |
|---|---|---|---|---|---|---|---|
|  |  |  |  |  |  | 是 / 否 | 升级 / 维持 / 降级 / 移除 |

## 十一、模块八：仓位建议系统

### 11.1 模块目标

给个人投资者一个“非喊单、非保证收益”的仓位参考框架。

重点不是告诉用户一定买什么，而是告诉用户：

- 当前应该偏进攻还是防守
- BTC、ETH、稳定币、Alpha 标的之间如何分配
- 在不同市场环境下，是否适合增加高风险仓位
- 小白用户如何避免一次性重仓和追高

### 11.2 仓位建议的输入

| 输入 | 说明 |
|---|---|
| BTC 周期位置 | 判断 BTC 是底部、中期、过热还是顶部风险 |
| 今日市场环境 | 强进攻 / 中性轮动 / 谨慎 / 防守 |
| 稳定币趋势 | 场内资金是否扩张 |
| ETF / DAT 流入 | 外部资金是否支持风险资产 |
| 强链强赛道质量 | 是否有明确结构性机会 |
| Alpha 观察池质量 | A/B/C/D 项目占比 |
| 解锁与宏观风险 | 是否存在短期重大风险 |
| 用户风险偏好 | 保守型 / 平衡型 / 进攻型 |

### 11.3 默认用户画像

本产品默认用户为：

> 投资小白 + 少量资金 + 周期投资为主 + 不适合高杠杆 + 不适合重仓山寨。

因此默认仓位建议应偏保守。

### 11.4 仓位建议类型

#### 保守型

| 资产 | 强进攻 | 中性轮动 | 谨慎 | 防守 |
|---|---:|---:|---:|---:|
| BTC / ETH | 50-70% | 40-60% | 30-50% | 20-40% |
| 稳定币 | 20-40% | 30-50% | 50-70% | 60-80% |
| Alpha 标的 | 5-10% | 3-8% | 0-5% | 0-3% |
| 高风险热点 | 0-3% | 0-2% | 0 | 0 |

#### 平衡型

| 资产 | 强进攻 | 中性轮动 | 谨慎 | 防守 |
|---|---:|---:|---:|---:|
| BTC / ETH | 40-60% | 35-55% | 30-45% | 20-35% |
| 稳定币 | 20-35% | 30-45% | 45-60% | 60-75% |
| Alpha 标的 | 10-20% | 5-15% | 0-8% | 0-5% |
| 高风险热点 | 0-5% | 0-3% | 0-2% | 0 |

#### 进攻型

仅适合有经验用户，不作为默认推荐。

| 资产 | 强进攻 | 中性轮动 | 谨慎 | 防守 |
|---|---:|---:|---:|---:|
| BTC / ETH | 30-50% | 30-45% | 25-40% | 20-30% |
| 稳定币 | 10-30% | 25-40% | 40-60% | 60-80% |
| Alpha 标的 | 20-35% | 10-25% | 5-10% | 0-5% |
| 高风险热点 | 0-10% | 0-5% | 0-3% | 0 |

### 11.5 仓位建议输出格式

【今日仓位建议】

- 用户风险类型：保守型 / 平衡型 / 进攻型
- 当前市场环境：
- BTC 周期位置：
- 建议 BTC / ETH 占比：
- 建议稳定币占比：
- 建议 Alpha 标的占比：
- 建议高风险热点占比：
- 今日是否适合新增仓位：
- 今日是否适合只观察：
- 不建议操作：
- 仓位调整理由：
- 风险提示：

### 11.5.1 多资产仓位规则

系统不建议用户平均持有 30 个币。

30 个币只作为观察宇宙，不作为默认持仓组合。

默认仓位仍分为四类：

1. 核心仓：BTC / ETH
2. 防守仓：稳定币
3. 观察仓：Alpha 标的
4. 极小仓：高风险热点

主流币和热点币只有在满足以下条件时，才可从“观察资产”升级为“候选仓位”：

- 所属链或赛道正在变强
- 项目数据连续改善
- 代币价值传导清晰
- 估值没有明显透支
- 未来 30 / 90 / 180 天解锁风险可控
- 当前市场环境不是谨慎或防守

### 11.6 仓位纪律

系统应明确提示：

1. 不做满仓。
2. 不使用高杠杆。
3. 不因为单日上涨追高。
4. 不因为 KOL 热度买入。
5. Alpha 标的单个仓位不宜过大。
6. 如果市场环境转谨慎或防守，应优先保护本金。
7. 如果项目进入观察池，不代表马上买入。
8. 任何仓位建议都必须结合用户自身风险承受能力。

## 十七、最终产品价值

升级后的产品，不只是一个 Web3 数据看板，而是一个个人投资决策训练系统。

它帮助用户建立以下能力：

1. 看懂 BTC 周期位置。
2. 判断今天市场是否适合进攻。
3. 找到资金正在流入的链。
4. 找到数据正在变强的赛道。
5. 区分强协议和强代币。
6. 用 Alpha 雷达筛选观察池。
7. 跟踪历史观察池，避免只新增不复盘。
8. 根据市场状态做仓位管理。
9. 每天复盘自己的判断依据。
10. 少一点情绪化追涨，多一点数据化验证。

核心原则：

> 不追热点，而是追踪数据。  
> 不迷信叙事，而是验证现金流和用户行为。  
> 不把协议强等同于代币强。  
> 不把观察池等同于买入清单。  
> 不用仓位表达情绪，而用仓位表达风险判断。

## 十八、用户每日使用流程

### 18.1 模块目标

把系统从“数据展示工具”变成“每日投研决策流程”。

用户每天打开系统后，不需要先看所有数据，而是按固定顺序完成一次判断：

> 先判断市场环境，再判断 BTC 周期，再看稳定币和主流币异动，再看资金流向和观察池变化，最后决定今天是否需要新增风险或只做复盘。

### 18.2 每日 6 步使用流程

| 步骤 | 用户动作 | 系统输出 | 用户决策 |
|---|---|---|---|
| 第一步 | 查看首页决策卡 | 今日市场环境、总分、最大风险 | 判断今天能不能进攻 |
| 第二步 | 查看 BTC 周期卡 | BTC 周期阶段、操作倾向 | 判断是否适合加 BTC / 持有 / 减仓 / 观望 |
| 第三步 | 查看多资产观察池 | 稳定币流动性、ETH/主流资产状态、主流币异动 Top 5 | 判断 30 个币里哪些只是价格波动，哪些值得观察 |
| 第四步 | 查看强链强赛道 | 今日资金和交易活跃方向 | 判断资金流向哪里 |
| 第五步 | 查看 Alpha 观察池 | 新增、升级、维持、降级、移除项目 | 判断哪些项目继续跟踪，哪些放弃 |
| 第六步 | 填写复盘问题 | 今日判断依据、风险、明日验证点 | 训练自己是否按系统决策，而不是被情绪带走 |

### 18.3 每日使用规则

1. 市场环境为“谨慎”或“防守”时，默认不新增高风险山寨仓位。
2. BTC 周期处于“过热区”或“顶部风险区”时，不因为山寨短线强势而重仓。
3. 观察池项目不等于买入清单，必须经过数据验证和价格回调确认。
4. 当项目价格上涨但协议数据没有同步改善时，优先标记为“价格先行，需谨慎”。
5. 每天至少回答一个复盘问题，避免系统只变成看板，而不是训练工具。

首页从上到下展示：

## 二十、评分规则细化

### 20.1 每日市场环境评分细化

#### Stablecoins 评分

| 条件 | 分数 | 判断 |
|---|---:|---|
| 7D > +0.5%，且 30D > +1% | +1 | 场内现金池扩张 |
| 7D 在 -0.5% 到 +0.5% 之间 | 0 | 场内资金横盘 |
| 7D < -0.5%，或 30D < -1% | -1 | 场内现金池收缩 |

#### DeFi TVL 评分

| 条件 | 分数 | 判断 |
|---|---:|---|
| 7D > +2%，且 30D > +5% | +1 | 链上资金沉淀增强 |
| 7D 在 -2% 到 +2% 之间 | 0 | TVL 横盘 |
| 7D < -2%，或 30D < -5% | -1 | 链上资金撤退 |

#### DEX Volume 评分

| 条件 | 分数 | 判断 |
|---|---:|---|
| 7D > +10%，且多个链同步增长 | +1 | 链上交易活跃度提升 |
| 7D 在 -10% 到 +10% 之间 | 0 | 交易热度中性 |
| 7D < -10% | -1 | 链上交易降温 |

#### ETF / DAT 评分

| 条件 | 分数 | 判断 |
|---|---:|---|
| BTC / ETH ETF 连续 3 日净流入，或单日明显大额流入 | +1 | 外部资金支持风险资产 |
| 流入流出混杂，无明显方向 | 0 | 资金方向不明确 |
| 连续 3 日净流出，或单日大额流出 | -1 | 外部资金撤退 |

#### 宏观政策评分

| 条件 | 分数 | 判断 |
|---|---:|---|
| 降息预期上升，DXY / 美债收益率回落，无重大监管利空 | +1 | 宏观环境偏支持风险资产 |
| 宏观数据未超预期，市场等待事件落地 | 0 | 宏观中性 |
| CPI / PCE 超预期，DXY / 美债收益率上行，或出现监管利空 | -1 | 宏观压制风险偏好 |

### 20.2 Perps 风险修正规则

Perps 不直接纳入 +1/-1 总分，但作为风险修正项。

| 条件 | 风险判断 | 系统动作 |
|---|---|---|
| Perps Volume 上升，OI 温和上升，Funding 中性 | 健康活跃 | 可继续观察强 Perps 协议 |
| Perps Volume 上升，OI 快速上升，Funding 偏高 | 杠杆升温 | 首页提示“追高风险” |
| Volume 下降，OI 上升 | 风险偏高 | 提醒可能存在杠杆堆积 |
| Volume 和 OI 同时下降 | 投机降温 | 降低 Perps 赛道评分 |

### 20.3 Alpha 评分细化

| 评分项 | 0-3 分 | 4-6 分 | 7-8 分 | 9-10 分 |
|---|---|---|---|---|
| 所属链 | 链上资金流出，交易低迷 | 单项指标改善 | TVL / DEX / Fees 至少两项改善 | TVL、稳定币、DEX、Fees 多项共振 |
| 所属赛道 | 只有叙事，无数据 | 单项目异动 | 多项目同步改善 | 多项目共振 + 资金/政策/产品催化 |
| 协议数据 | TVL / Fees / Revenue 均弱 | 单项数据改善 | 两项以上连续改善 | TVL、Volume、Fees、Revenue 同步改善 |
| 收入质量 | 补贴、刷量或短期激励为主 | 收入来源不够清晰 | 有真实用户付费 | 无明显补贴，收入稳定且可持续 |
| 代币价值传导 | 纯治理，暂无捕获 | 有潜在预期 | 有 staking / buyback / burn 之一 | 有明确现金流、回购或持有人收入 |
| 估值合理性 | FDV 明显透支 | 估值偏高但可跟踪 | 与同赛道接近 | 增长和估值匹配，未明显透支 |
| 供给结构 | 大额解锁临近 | 有中等释放压力 | 解锁可控 | 高流通、低通胀、短期无大额解锁 |
| 流动性结构 | 流动性差，难进出 | 成交量不足 | CEX/DEX 流动性尚可 | 深度充足，成交量与市值匹配 |
| 催化因素 | 无明确催化 | 叙事催化为主 | 有产品/生态/上币催化 | 多个可验证催化叠加 |

### 20.4 评级动作映射

| 综合分 | 评级 | 默认动作 |
|---:|---|---|
| 80+ | A | 重点研究，不代表马上买入，等待价格和风险确认 |
| 65-79 | B | 进入观察池，等待数据继续验证 |
| 50-64 | C | 低优先级观察，不主动投入时间 |
| 50 以下 | D | 暂时排除或仅事件触发时查看 |

| 宏观数据 | FRED / Trading Economics | Investing / CME FedWatch | CPI、PCE、利率、DXY、美债收益率 |
| 社媒热度 | Kaito / LunarCrush | X / Google Trends | 只作为辅助，不作为买入依据 |
| 鲸鱼行为 | Arkham / Nansen | Etherscan / Solscan | 必须判断是否为转交易所、做市、套利或 OTC |

### 21.3 口径冲突处理规则

1. 大盘和链上主口径优先 DefiLlama。
2. 代币价格、市值、FDV 优先 CoinGecko。
3. 收入类数据必须拆分为 Fees、Revenue、Holder Revenue，不能混用。
4. 解锁数据必须标明来源、日期、解锁对象和占流通比例。
5. 若两个主流数据源差异超过 10%，报告中必须提示“口径差异”。
6. 若关键数据无法获取，填写 N/A，不允许自动编造或用旧数据替代。
7. 若数据源显示异常 spike，需要结合 7D / 30D 趋势判断，不能只用单日数据。
8. 所有结论必须区分“事实数据”和“系统推断”。

### 21.4 数据更新时间规则

| 数据类型 | 建议更新频率 | 说明 |
|---|---|---|
| BTC / ETH 价格 | 每日 | 日报级别不需要实时分钟级 |
| Stablecoins / TVL / DEX / Perps | 每日 | 用于市场环境评分 |
| Fees / Revenue | 每日或每 2 日 | 对强协议和 Alpha 观察池重要 |
| 解锁 | 每周 + 事件触发 | 30天内解锁需要重点提醒 |
| BTC 周期指标 | 每周 2-3 次 | 不必每天过度解读 |
| 宏观事件 | 每日检查事件日历 | CPI、PCE、FOMC、非农需要重点标记 |
| 观察池评级 | A 类每日，B 类 2-3 日，C 类每周 | 避免每天过度交易 |

### 21.5 数据新鲜度标识规则

所有影响首页决策卡、市场评分、Alpha 评级和仓位建议的关键数据，都需要展示“数据时间”。

| 状态 | 触发条件 | 页面提示 | 是否参与评分 |
|---|---|---|---|
| 正常 | 在预期更新时间内完成更新 | 正常显示数据时间 | 是 |
| 延迟 | 超过预期更新时间的 1.5 倍 | 黄色提示：数据可能过时 | 谨慎参与评分 |
| 不可用 | 超过预期更新时间的 3 倍或 API 失败 | 红色提示：数据不可用，使用缓存值或 N/A | 默认不参与核心评分 |
| 口径冲突 | 主数据源与备用数据源差异超过 10% | 提示口径差异，并展示来源 | 需人工确认 |
| 手动覆盖 | 用户人工修改关键数据 | 标记“已手动覆盖” | 可参与评分，但必须记录原因 |

### 21.6 异常值处理规则

当单日变化明显异常时，系统不得直接把异常值纳入评分。

异常情况包括：

1. TVL / Volume / Fees 单日变化超过历史常规波动区间。
2. 某协议因迁移、合并口径或补贴活动导致数据暴增。
3. 价格上涨但链上数据没有同步改善。
4. 主数据源和备用数据源出现明显冲突。

系统动作：

- 标记为“异常波动，需人工确认”。
- 暂不自动提升 Alpha 评级。
- 用户可选择“确认有效 / 标记无效 / 等待观察”。
- 所有人工确认必须记录原因。

### 21.7 手动覆盖规则

V1.2 允许用户手动修正关键数据，但必须保留痕迹，避免事后无法复盘。

| 字段 | 要求 |
|---|---|
| 覆盖前数值 | 自动采集或原始录入值 |
| 覆盖后数值 | 用户修改后的数值 |
| 覆盖原因 | 必填，如“数据源延迟”“项目迁移导致 TVL 异常” |
| 覆盖人 | 默认当前用户 |
| 覆盖时间 | 自动记录 |
| 是否影响评分 | 是 / 否 |

## 二十二、观察池生命周期管理

### 22.1 模块目标

把 Alpha 观察池从“项目列表”升级成“投研 CRM”，让每个项目都有明确状态和下一步动作。

### 22.2 生命周期状态

| 状态 | 含义 | 用户动作 |
|---|---|---|
| 新发现 | 数据刚异动，首次进入观察 | 不操作，只记录原因 |
| 观察中 | 数据有改善，但验证不充分 | 等待 3-7 天数据确认 |
| 重点跟踪 | 多项数据共振，风险可控 | 研究代币、估值、解锁和价格位置 |
| 风险升高 | 价格涨太快、解锁临近、数据转弱 | 降低优先级或等待风险释放 |
| 移除 | 原始逻辑失效或风险不可控 | 从观察池移出 |

### 22.3 评级与生命周期的区别

| 概念 | 作用 | 示例 |
|---|---|---|
| 评级 A/B/C/D | 表示项目质量 | A = 强协议 + 强传导 + 风险可控 |
| 生命周期状态 | 表示当前处理动作 | 重点跟踪 / 风险升高 / 移除 |

示例：

- HYPE：评级 A-，状态可为“重点跟踪”。
- PUMP：评级 B，状态可能是“风险升高”。
- MORPHO：评级 C+，状态可为“观察中”。

### 22.4 状态流转规则

```text
新发现 → 观察中 → 重点跟踪 → 继续跟踪 / 风险升高 / 移除
                 ↓
              风险升高 → 观察中 / 移除
```

### 22.5 观察池字段补充

| 字段 | 说明 |
|---|---|
| 生命周期状态 | 新发现 / 观察中 / 重点跟踪 / 风险升高 / 移除 |
| 状态更新时间 | 最近一次调整状态的日期 |
| 原始加入理由 | 当时为什么进入观察池 |
| 当前是否仍成立 | 是 / 否 / 部分成立 |
| 升级条件 | 哪些数据改善后升级 |
| 降级条件 | 哪些数据恶化后降级 |
| 移除条件 | 哪些风险触发后移除 |
| 最迟复查日期 | 防止项目长期无人处理 |
| 人工备注 | 用户主观判断与研究记录 |

## 二十三、风险预警系统

### 23.1 模块目标

本系统的核心价值不是“发现更多币”，而是帮助用户少一点情绪化追涨。因此风险预警应放在首页、观察池页和仓位建议页。

### 23.2 预警类型

| 预警类型 | 触发条件 | 提醒内容 | 系统动作 |
|---|---|---|---|
| 市场防守预警 | 市场总分 ≤ -3 | 今日进入防守，不新增高风险仓位 | 首页红色提示 |
| 谨慎预警 | 市场总分 -1 到 -2 | 今天只观察，降低新增风险 | 仓位建议偏保守 |
| BTC 顶部风险预警 | MVRV、NUPL、恐惧贪婪等多项过热 | BTC 可能进入顶部风险区 | 提醒分批降低风险敞口 |
| 杠杆过热预警 | Perps OI 快速增加，Funding 偏高 | 防止追高后被清算 | 降低 Perps 赛道评分 |
| 解锁预警 | 观察池项目 30 天内大额解锁 | 注意供给压力 | 项目标记“风险升高” |
| 代币传导弱预警 | Fees 增长但 Holder Revenue 为 0 | 强协议不等于强代币 | 降低代币传导评分 |
| 估值透支预警 | FDV/Revenue 远高于同赛道 | 当前价格可能已反映过度乐观 | 降低估值合理性评分 |
| 价格先行预警 | Token 7D 涨幅大，但 TVL/Fees/Revenue 无改善 | 价格先行，需谨慎 | 标记为“风险升高” |
| 数据失效预警 | 原始加入观察池理由不再成立 | 项目逻辑失效 | 降级或移除 |
| 数据新鲜度预警 | 关键数据延迟、缺失或口径冲突 | 今日判断可信度下降 | 首页提示并要求人工确认 |
| 链下尽调预警 | 团队、审计、合约权限、合规出现未通过项 | 链下风险可能推翻链上数据 | 降级、暂停观察或移除 |
| 安全事件预警 | 协议被攻击、合约暂停、审计风险暴露 | 一票否决风险 | 立即标记为高风险 |
| 监管风险预警 | 项目或赛道出现重大监管事件 | 风险偏好下降 | 降低赛道评分 |

### 23.3 预警优先级

| 优先级 | 类型 | 处理方式 |
|---|---|---|
| P0 | 安全事件、严重监管、流动性枯竭 | 立即提醒，项目暂停观察或移除 |
| P1 | 大额解锁、估值明显透支、BTC 顶部风险 | 首页强提醒，降低仓位倾向 |
| P2 | 杠杆升温、价格先行、代币传导弱 | 项目标记风险，继续观察 |
| P3 | 数据缺失、口径差异、社媒异常热度 | 报告中提示，不作为单独决策依据 |

### 23.4 预警展示位置

1. 首页决策卡：展示今日最大 1-3 个风险。
2. Alpha 观察池页：每个项目展示风险标签。
3. 仓位建议页：根据预警自动调整仓位倾向。
4. 飞书记录：每日记录风险触发原因。

## 二十四、人工输入与个人复盘区

### 24.1 模块目标

系统不能完全替代用户判断，尤其是项目叙事、官方文档变化、个人操作原因和主观偏见，需要人工记录。

### 24.2 人工输入字段

| 字段 | 说明 |
|---|---|
| 人工备注 | 自己对项目的理解和疑问 |
| 是否已读官方文档 | 是 / 否 |
| 是否看过 Tokenomics | 是 / 否 |
| 是否确认代币价值传导 | 是 / 否 / 不确定 |
| 我为什么关注它 | 防止只是因为涨了才关注 |
| 我不会买它的原因 | 反向约束，避免 FOMO |
| 需要验证的数据 | 明天或下周重点看什么 |
| 最迟复查日期 | 防止观察池堆积 |
| 是否已经操作 | 是 / 否 |
| 操作原因 | 数据驱动 / 情绪驱动 / 外部影响 / 其他 |
| 是否符合系统建议 | 是 / 否 / 部分符合 |

### 24.2.1 Alpha 项目尽调清单

每个进入“重点跟踪”的 Alpha 项目，至少需要完成以下人工检查。MVP 阶段只做模板和记录，不接入自动化尽调 API。

| 尽调项 | 检查内容 | 通过标准 | 不通过后的系统动作 |
|---|---|---|---|
| 团队背景 | 团队是否公开、过往项目是否可信 | 至少有公开成员或可信背书，无严重负面记录 | 标记高风险，不进入重点跟踪 |
| 融资与投资方 | 是否有可信投资方、估值是否异常 | 融资信息可查，估值与阶段基本匹配 | 降低评级或仅观察 |
| 审计状态 | 是否经过审计，是否有未修复高危漏洞 | 已审计且无重大未修复漏洞 | 暂停观察或移除 |
| 合约权限 | 是否存在单地址权限、无限 mint、冻结、黑名单等 | 多签、时间锁、权限透明 | 一票否决或高风险标记 |
| 社媒健康度 | X / Discord / Telegram 是否真实活跃 | 互动正常，无明显刷量 | 仅作为辅助，不提升评级 |
| 法律合规 | 是否涉及重大监管诉讼或证券化争议 | 无明显重大监管风险 | 降低赛道或项目评分 |

尽调结论规则：

1. 任一“一票否决”项触发，项目不得进入重点跟踪。
2. 多个“待验证”项存在时，项目只能保持观察中。
3. 链下尽调通过不代表可以买入，只代表没有明显一票否决风险。
4. 尽调结果必须进入项目详情页和每日 / 每周复盘记录。

## 二十五、MVP 验收标准

### 25.1 模块目标

明确第一版做到什么程度算完成，避免产品需求无限膨胀。

### 25.2 MVP 范围

MVP 只做以下 9 个核心能力：

1. BTC 周期记录与判断
2. 每日市场环境评分
3. 多资产观察池基础扫描
4. Alpha 观察池管理
5. 仓位建议输出
6. 每日复盘记录
7. 数据来源、更新时间和数据状态标识
8. 代币价值传导标签与人工判断
9. Alpha 项目链下尽调清单

### 25.3 暂不纳入 MVP 的功能

| 功能 | 暂缓原因 |
|---|---|
| 自动交易 | 风险高，不符合产品定位 |
| 实时分钟级行情 | 容易诱导短线交易 |
| 多用户系统 | 当前以个人使用为主 |
| 复杂回测 | 第一阶段先验证投研流程 |
| 付费数据源 | 先用免费或低成本工具跑通 |
| Kaito / Nansen 深度接入 | 成本高，作为二期事件触发补充 |
| 自动推送机器人 | 先完成日报和观察池沉淀 |
| 多地址 / 多钱包自动聚合 | 会把产品变成资产管理系统，V1.2 先不做 |
| 自动交易记录与盈亏统计 | 需要交易数据和隐私设计，放到 V2.0 |
| 复杂回测引擎 | 需要至少 30 天以上有效历史快照，先预留字段 |
| 个性化评分权重滑块 | 容易增加配置复杂度，第一版先用默认模型 |
| 自动链下尽调 API | 先用人工清单，后续再接入第三方信誉数据 |
| Telegram / Discord / Web Push 实时推送 | 先完成移动端可读和日报沉淀，推送放 V1.5 |

## 二十六、V1.2 补充后的产品原则

升级到 V1.2 后，本系统的核心原则进一步明确为：

1. **先判断环境，再判断标的。** 任何项目都不能脱离市场环境和 BTC 周期单独判断。
2. **先入观察池，再考虑操作。** 观察池不是买入清单，而是验证清单。
3. **30 个币是观察宇宙，不是持仓清单。** BTC 负责周期，ETH 代表主流风险资产，稳定币代表资金水位，主流币和热点币只负责提供观察线索。
4. **先看风险，再看收益。** 首页必须展示最大风险，避免用户只看机会。
5. **先验证数据，再相信叙事。** 社媒热度、KOL 推荐和价格上涨都必须由链上数据验证。
6. **先沉淀记录，再追求准确。** MVP 阶段的目标不是预测每一天，而是建立可复盘的判断框架。
7. **用仓位表达风险判断，而不是表达情绪。** 市场不支持进攻时，即使有强项目，也只做观察或小仓验证。
8. **自动采集，但永远可干预。** 系统可以自动化，但必须承认数据源不完美，并提供人工修正入口。
9. **代币传导量化，但不伪精确。** MVP 先用标签、强弱判断和人工依据，避免用缺失数据包装成精确公式。
10. **链下尽调补齐链上盲区。** 团队、合约权限、审计和合规风险可以直接推翻链上数据结论。
11. **移动端先可读，不急着 App 化。** 手机端优先展示今日决策和风险，不做复杂推送和实时交易体验。
12. **路线图分层，不把未来能力塞进 MVP。** 多钱包、回测、推送、个性化权重属于后续迭代，不影响第一版投研闭环。

> V1.2 的目标不是让用户每天买到涨得最多的币，而是让用户每天少一点瞎买、多一点验证、多一点复盘，并用 30 币种池建立可控的观察宇宙。
